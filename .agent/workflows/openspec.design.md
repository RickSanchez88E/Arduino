# /openspec.design

为提案写一份轻量设计文档 — 够开始编码即可，不追求穷尽。

## 设计原则

- ✅ 写「怎么做」和「改哪些文件」
- ✅ 列出可能踩坑的地方
- ❌ 不要求完整的架构图
- ❌ 不要求覆盖所有边界条件

## 步骤

1. **读取上下文**
   - 读取 `docs/openspec/OS-{NNN}-{slug}/proposal.md`
   - 读取 `docs/project-context/current-context.md`
   - 如果提案涉及已有代码，浏览相关源码文件

2. **撰写设计文档**
   基于 `docs/openspec/_templates/design.md` 创建:
   ```
   docs/openspec/OS-{NNN}-{slug}/design.md
   ```
   
   重点覆盖:
   - **做法概要**: 用 1-3 段话说明实现思路
   - **文件清单**: 需要新建/修改/删除的文件
   - **接口变更**: 如果有 API 或数据结构变更
   - **已知风险**: 可能出问题的地方
   - **绕不开的约束**: 来自宪法或现有架构的限制

3. **宪法快速检查**
   只检查三项:
   - 是否超出 5 文件修改限制？（如果是 → 建议分拆或升级到 Spec Kit）
   - 是否涉及安全敏感区域？
   - 是否需要新增依赖？

4. **更新提案状态**
   更新 `proposal.md` 状态为 `📐 已设计`。

5. **输出设计摘要 (中文)**
   用平实的中文说明打算怎么做。

> [!TIP]
> 设计文档不需要完美。它的目标是「够动手」且「可回顾」。
