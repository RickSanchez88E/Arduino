# 项目宪法 (Project Constitution)

> 版本: 1.0
> 生效日期: 2026-03-13
> 状态: ✅ 生效中 (Active)

---

## 不可违反原则 (Non-Negotiable Principles)

以下原则适用于所有 agent 和人工操作。违反任何原则需要明确的书面批准。

---

### 原则 1: 架构一致性 (Architecture Consistency)

> **所有代码变更必须符合已批准的架构设计。**

- 不得引入未在架构图中声明的新层次或服务
- 新模块必须遵循既定的目录结构和命名惯例
- 跨层调用必须通过定义的接口进行
- 技术栈变更需要正式的 ADR (Architecture Decision Record)

```
违规示例:
❌ 在 UI 层直接操作数据库
❌ 引入新的状态管理库而未提交 ADR
❌ 创建不符合既定模式的新目录结构
```

---

### 原则 2: 测试要求 (Testing Requirements)

> **所有功能代码必须有对应的测试覆盖。**

- 新功能必须包含单元测试
- 核心业务逻辑（工作流引擎）必须有集成测试
- API 端点必须有端到端测试
- 测试覆盖率不得低于已定义的阈值
- Bug 修复必须附带回归测试

| 测试类型 | 最低覆盖要求 | 适用范围 |
|----------|-------------|----------|
| 单元测试 | 80% | 所有业务逻辑模块 |
| 集成测试 | 关键路径覆盖 | 工作流引擎、数据持久化 |
| E2E 测试 | 核心流程覆盖 | 用户可见的主要操作 |

---

### 原则 3: 禁止未经批准的大规模重构 (No Broad Refactors Without Approval)

> **影响 3 个以上文件的重构必须事先获得明确批准。**

- 重构提案必须以 spec 形式提交，包含：
  - 影响范围分析
  - 风险评估
  - 回滚计划
  - 预期收益量化
- 自动化 agent 在单次会话中修改不超过 5 个文件
- 跨模块重构必须分阶段执行

---

### 原则 4: 性能边界 (Performance Boundaries)

> **所有变更不得导致可衡量的性能退化。**

- 页面初始加载时间不超过 3 秒 (3G 网络模拟)
- 工作流渲染不超过 100ms（100 节点以内）
- API 响应时间 P95 不超过 500ms
- 打包体积增长不超过 10% 需要说明理由
- 禁止引入已知的性能反模式（如 N+1 查询、无限循环事件）

---

### 原则 5: 安全约束 (Security Constraints)

> **安全是不可协商的底线。**

- 所有用户输入必须进行验证和清理
- 敏感数据（密钥、令牌）禁止硬编码或提交到版本控制
- 依赖更新必须检查已知漏洞 (CVE)
- 认证/授权变更需要安全审查
- 工作流执行必须在沙箱环境中运行
- 遵循最小权限原则

```
必须检查项:
☑ 无硬编码密钥
☑ 输入验证
☑ SQL 注入防护
☑ XSS 防护
☑ CSRF 防护
☑ 依赖漏洞扫描
```

---

### 原则 6: 文档更新要求 (Documentation Update Requirements)

> **代码变更必须同步更新相关文档。**

- API 变更必须更新接口文档
- 架构变更必须更新架构图
- 新功能必须更新用户文档
- 配置变更必须更新部署文档
- 所有用户可见的摘要和文档必须提供中文版本

---

### 原则 7: 规范驱动开发 (Spec-Driven Development)

> **先写规范，再写代码。没有规范的代码将被拒绝。**

本项目支持两条并行的规范流程:

**Spec Kit (正式流程)** — 适用于大功能、架构变更:
- 每个功能/变更必须有对应的 spec 文件
- Spec 必须包含：目标、范围、技术方案、测试策略、验收标准
- Spec 必须在实施前获得批准
- 实施完成后 spec 状态必须更新

```
Spec Kit 流程:
specify → 审批 → plan → tasks → implement → handoff
```

**OpenSpec (轻量流程)** — 适用于 bug 修复、小迭代、配置调整:
- 每个变更必须有对应的 proposal 文件
- Proposal 需回答: 做什么、为什么、改哪里
- 无硬性审批门，但高风险改动建议升级到 Spec Kit
- 完成后必须创建 archive-note（含合并风险）

```
OpenSpec 流程:
propose → [clarify] → design → tasks → apply → archive
```

---

## 执行与例外 (Enforcement & Exceptions)

- 所有 agent 在每次会话开始时必须阅读本宪法
- 违反原则的变更将被标记并回滚
- 例外情况需要用户明确的书面批准
- 紧急修复（P0 级别），可以事后补充文档，但必须在 24 小时内完成
